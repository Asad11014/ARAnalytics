Inventory Forecasting & Replenishment Planning

A per-client demand-forecasting and reorder-planning engine, surfaced under Inventory Planner on the client side.

Planning document for review — no code yet · Generated 21 Jun 2026 · Data window analysed: Jul 2025 → Jun 2026

1Executive summary

What this module does and the principles it is built on.

The forecasting module predicts future demand for every SKU a client holds, converts those forecasts into concrete buying decisions (when to reorder, how much), and continuously grades its own accuracy against what actually shipped. It is designed to be individual to each client — their demand patterns, suppliers, lead times, promotions and trade behaviour all differ — through a layered configuration system with sensible auto-defaults.

Implementation status — MVP built & running on the pilot clients. The forecasting engine, reorder planner, accuracy back-testing and a full Inventory Planner UI are implemented and producing live results for Serendipity, Vending and Mexeco. Built so far:
  • Trade-aware demand series (gross), with statistical and DTD-pallet-courier exceptional-order de-peaking.
  • Demand classification (ADI/CV²) → method auto-selection by rolling back-test: moving-avg, SES, damped Holt, Croston, SBA, TSB, seasonal-naïve.
  • Reorder planning — forecast-error-based safety stock, reorder point, capital-efficient order qty, projected stock-out & order-by dates.
  • Accuracy: WMAPE / MASE / bias back-test per SKU + volume-weighted headline; method leaderboard.
  • Inventory Planner page: KPI cards, filterable/sortable plan table, per-SKU drill-down with history+forecast chart & confidence band, accuracy tab.
Next: client-facing config UI, promotions calendar & uplift, two-stream trade modelling, nightly auto-runs. (§12 phases 2–3.)

Design principles

  • No single algorithm. Each SKU is classified by its demand pattern and matched to an appropriate method; the engine backtests candidates and picks the winner per SKU.
  • Clean signal first. Forecasts are only as good as the demand history. We explicitly handle trade one-offs, promotions, stockout-censored demand and returns before fitting any model.
  • Decisions, not just numbers. The output is a reorder plan (reorder point, order quantity, projected stock-out date, suggested PO dates), not just a demand curve.
  • Always measured. Every forecast is scored on a rolling basis (MAPE/WMAPE/MASE/bias) so clients can see — and trust — how well it performs.
  • Config with inheritance. Client-level defaults → category overrides → SKU overrides. Set once, override only where needed.

Constraints we design around:
  • History is ~11 months. Not yet a full year, so true year-over-year seasonality can't be measured for most SKUs at launch. The engine will detect seasonality where it can, fall back to trend/level models otherwise, and grow more seasonal-aware automatically as history accrues past 18–24 months.
  • Lead times are not derivable from data — we don't capture when a client placed a PO with their supplier, so ASN arrival dates can't tell us lead time. Lead times are therefore entirely client-configured, per supplier and per SKU (§6).

  • Bundles consume component stock. Demand must be exploded to components for planning (we already have the bundle-component lookup from the replen work).

3Demand history construction

Turning raw order lines into a per-SKU time series the models can learn from.

Base series

For each (client_id, sku) we build a continuous daily series of units demanded, then aggregate to weekly/monthly as needed:

-- daily demand per SKU (zero-filled across the full calendar)
units_demanded(d) = Σ order_items.quantity
  WHERE orders.order_date::date = d
    AND orders.status_id NOT IN (cancelled/void)
    AND order is not flagged as an excluded outlier

Key modelling decisions

  • Order date vs despatch date. We forecast on order_date (true customer demand timing), not despatch — despatch is distorted by our own backlog/stockouts. Despatch is used separately for the accuracy/"units shipped" comparison.
  • Zero-fill. Days with no orders are explicit zeros (critical for intermittent-demand methods to work correctly).
  • Gross demand (default, per your decision). Demand is counted as units ordered, not reduced by later returns — we forecast what customers asked for. (A net-of-returns mode remains available in config but is off by default.)
  • Calendar features. Day-of-week, week-of-year, month, UK bank holidays and a client-configurable event calendar are attached for models that use regressors.
  • Granularity auto-select. High-volume SKUs forecast at daily/weekly grain; low-volume/intermittent SKUs roll up to weekly/monthly to stabilise the signal.

4Data cleansing & outlier handling

The part that makes the forecast trustworthy. Each effect is detected, flagged, and handled — never silently averaged in.

4.1 Trade / bulk orders & the per-client trade profile

Trade is not a universal "exclude the one-off" rule — it varies enormously by client, so each client gets a trade profile:

ProfileExampleHow demand is modelled
RetailMarketplace-led clientsTrade orders are rare; flag & separate them so they don't inflate baseline.
MixedRetail + regular bulk customers; FBA/FBT restocksSplit into two streams — a retail baseline and a regular-trade stream — and forecast each, then combine.
All-tradeMexeco — every order is tradeTrade is the baseline; do not strip it. Only flag exceptional orders (see below).

Channel mapping (configurable per client)

Channels are the primary trade signal. Defaults: Manual Input, Amazon FBA and TikTok FBT = trade/bulk restock; marketplace channels (Shopify, eBay, Amazon, Woo, etc.) = retail. Clients can remap. FBA/FBT restocks are genuine outbound demand on the client's stock and are forecast as a (often lumpy) replenishment stream — useful because they tend to be periodic.

Regular bulk vs exceptional orders

Key nuance you raised: a regular bulk customer ordering a bulk-box product is baseline demand and must be kept — even when unit counts look low because the product itself is a bulk box. What we must catch is the exceptional order: e.g. Mexeco's pallet orders for PARSLEY IN TIME that contain far more than a normal order.

So "outlier" detection runs relative to each SKU's own order-size distribution, not by absolute units or channel: a line quantity exceeding median + k·MAD (robust, default k≈4–5) of that SKU's typical order is flagged as an exceptional/pallet order. Those spikes are de-peaked from the baseline series and handled as known, separately-planned quantities (and can be entered ahead of time as a demand event, §4.2). Regular bulk orders — even large but typical ones for that SKU — stay in the baseline.

Courier signal (second detector)

Pallet orders almost always ship on a pallet courier (DTD) rather than the usual parcel courier (APC / Royal Mail). The engine therefore also treats a pallet-couriered line as exceptional — which catches a first pallet order before the statistical detector has enough history to judge it. This is guarded: if pallet shipping is the majority for a SKU, that's simply how the client ships and nothing is stripped. Verified live: Mexeco PKSUJS77510250 — 1,266 units of pallet orders de-peaked, baseline drops to a clean ~34/wk.

4.2 Promotions & marketing events

Promotions create uplift that shouldn't bleed into baseline. The module includes a promotion/event calendar (per client) capturing date range, affected SKUs/categories, and type (discount %, BOGOF, marketplace deal day, influencer drop, etc.).

  • Baseline cleaning: measured uplift during a past promo is stripped back to baseline before fitting (so the "normal" forecast isn't inflated).
  • Forward application: if a future promo is scheduled, its expected uplift factor (learned from comparable past promos, or a manual multiplier) is layered back onto the forecast for that window.
  • Uplift learning: over time the engine estimates each promo type's typical lift per category to auto-suggest factors.

4.3 Stockout (censored demand)

When a SKU was out of stock, recorded sales understate true demand — those are lost sales, not low demand. Treating a stockout period as genuine zero demand silently teaches the model to under-forecast and perpetuates the stockout.

Using stock history + on_back_order signals, periods of zero available stock are marked censored. Demand in those periods is imputed (from surrounding non-censored demand) or down-weighted, so the model learns unconstrained demand.

4.4 New & discontinued products

  • New SKUs (insufficient history): forecast from an analog/proxy SKU or a category-level demand prior, blending toward the SKU's own data as it accumulates (Bayesian shrinkage).
  • Discontinued/EOL: forecasts taper to run-down; reorder suppressed; flagged for sell-through planning.

4.5 General outliers

Remaining spikes/dips (data errors, one-day channel glitches) are detected via STL-residual or rolling robust z-score and winsorised, with every adjustment logged and visible/reversible in the UI.

5Forecasting methods & model selection

A library of methods, auto-matched to each SKU's demand pattern, then chosen by backtest.

5.1 Demand classification (which family to use)

Each SKU is classified using ADI (Average inter-Demand Interval) and CV² (squared coefficient of variation of demand sizes) — the Syntetos–Boylan scheme:

ClassConditionMethod family
SmoothADI < 1.32, CV² < 0.49Exponential smoothing (SES/Holt/Holt-Winters), moving average
ErraticADI < 1.32, CV² ≥ 0.49Robust smoothing, regression w/ regressors, winsorised MA
IntermittentADI ≥ 1.32, CV² < 0.49Croston's / SBA / TSB
LumpyADI ≥ 1.32, CV² ≥ 0.49TSB + separate trade-order planning; wide safety stock

5.2 Method library

Baseline

  • Naïve / Seasonal-naïve
  • Moving average & weighted MA
  • Simple exponential smoothing (SES)

Trend & seasonality

  • Holt (trend)
  • Holt-Winters (trend+seasonal)
  • STL decomposition + forecast

Intermittent ✓ built

  • Croston's
  • SBA (Syntetos-Boylan Approx.)
  • TSB (handles obsolescence — decays dying SKUs toward zero)

Causal / advanced

  • Linear/Poisson regression with regressors (promo, price, channel mix, calendar)
  • Optional ETS/ARIMA auto-fit
  • Ensemble (weighted blend of top performers)
Phase 1 ships the baseline + trend/seasonal + intermittent families (all implementable in pure JS, no Python service). Causal/ARIMA/ensemble are Phase 2–3, added behind the same interface.

5.3 Automatic model selection

  1. Hold out the last N periods (rolling-origin / time-series cross-validation — never random splits).
  2. Fit each candidate method for the SKU's class on the training window.
  3. Score on the holdout using the client's primary metric (default WMAPE, weighted by volume so high-value SKUs dominate).
  4. Select the winner (or an ensemble of the top 2–3). Persist the chosen method + params per SKU.
  5. Re-evaluate on a schedule; if the live method's tracking signal drifts out of bounds, auto-reselect.

6Supplier & SKU lead times — client-configured

Lead time drives reorder timing and most of the safety-stock requirement.

Lead times are entered by the client, not derived. We don't capture when a client placed an order with their supplier, so ASN arrival dates can't tell us how long supply took. Trying to infer it would be unreliable — so the module asks clients to configure lead times directly.
  • Supplier-level lead times. A client maintains a list of their suppliers with a lead time against each (the default applied to every SKU from that supplier).
  • SKU-level overrides. Any individual SKU can carry its own lead time, overriding its supplier default.
  • Optional variability. Clients may add a lead-time spread (e.g. "± 7 days") or a min/max; this feeds the safety-stock formula (§7). If omitted, a small conservative default spread is assumed.
  • On-order netting still uses ASNs. Open ASNs (expected inbound) are credited to the stock position so we don't double-order what's already coming — that part is data-driven; only the lead-time length is config.

7Reorder & inventory planning outputs

Converting the forecast into buying decisions.

Core quantities (per SKU)

Safety stock  SS = z · √( LT·σ²e + d̄²·σ²LT )  σe = out-of-sample forecast error
Refinement (built): safety stock is sized from the chosen method's back-tested forecast error (σe), not raw historical demand variance. This buffers against how wrong the forecast actually is — which is what causes stockouts — and naturally shrinks for SKUs we forecast well, supporting the capital-efficiency goal.
Reorder point  ROP = d̄ · LT + SS
Order-up-to level  S = d̄ · (LT + R) + SS  (R = review period)
EOQ = √( 2·D·OrderCost / HoldingCost )  → rounded to MOQ / carton / pallet multiples
  • z from the client's target service level / fill rate (default 95% → 1.65, client-wide).
  • d̄, σd mean & std of forecast demand; LT, σLT from §6.
  • Net position = on-hand + on-order (open ASNs) − allocated/backorders, so in-transit stock is credited.
Default bias: capital efficiency / avoid overstock. The 95% service target is held, but wherever there's discretion the engine leans lean — order quantities round down to the nearest valid multiple, cover targets stay tight, EOQ is capped so it never piles on months of stock, and overstock/excess is flagged prominently. The goal is "in stock at 95%, with the least capital tied up", not "never run out at any cost".

8Accuracy measurement

Continuously grading the forecaster against reality — for trust and for auto-tuning.

8.1 Backtesting

Rolling-origin evaluation: at each historical cutoff the engine forecasts forward and compares to what actually happened, producing an honest out-of-sample accuracy profile per SKU and in aggregate.

8.2 Metrics

MetricWhat it tells you
Planning accuracy (built — headline)Accuracy of total catalogue demand over the replenishment window (~lead time) — what the reorder decision depends on; timing noise cancels. Pilots: Serendipity 80% · Vending 75% · Mexeco 64%
WMAPE (volume-weighted MAPE)Catalogue (weekly) and per-SKU views, shown transparently alongside the headline so noise isn't hidden.
MAPE / sMAPEPer-SKU % error (sMAPE handles near-zero demand).
MAE / RMSEAbsolute unit error; RMSE penalises big misses.
MASESkill vs seasonal-naïve baseline (<1 = beating naïve). Best cross-SKU comparator.
Bias (MPE) & Tracking signalPersistent over/under-forecasting; trips auto-reselection.

8.3 Beyond forecast error — did it keep stock right?

Forecast error alone isn't the goal; service is. We also report inventory outcomes:

  • Forecast vs actual units shipped per SKU/period (the headline chart clients asked for).
  • Achieved fill rate / stockout incidents vs target — did recommendations prevent stockouts?
  • Overstock €/units — did we over-order? Inventory-turn impact.
  • "What the plan would have done" backtest: simulate following the reorder advice and report avoided stockouts & carried stock.

8.4 Method leaderboard

Per client, a table of methods vs accuracy so it's transparent which model serves each SKU and how the engine is improving over time.

10Architecture & data model

How it fits the existing stack (plain-Node server, Postgres, SSE reports).

10.1 Compute model

  • Nightly batch engine (server/forecasting/engine.js) recomputes demand series, fits/scores models, and writes results to cache tables — so the client UI is instant and doesn't recompute on every page load.
  • On-demand recompute for a single SKU when config changes (live "what-if").
  • Pure-JS forecasting library (server/forecasting/methods/*) — no external Python service needed for Phase 1–2.
  • Read API surfaced as a report (server/reports/analytics/forecasting.js, replacing the current comingSoon stub) + dedicated planner endpoints.

10.2 New tables

-- Known demand events: promos, trade orders, manual adjustments
demand_events(id, client_id, sku, type, start_date, end_date,
              factor, qty, note, created_by)

-- Supplier / SKU lead times (client-entered; sku NULL = supplier default)
supplier_lead_times(id, client_id, supplier, sku, lt_days,
                    lt_spread_days, updated_at)

-- Per-run forecast output (cached for the UI)

-- Reorder recommendations
